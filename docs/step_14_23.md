# ステップ 14.23: キャスト式に対する `expr_type()` の修正

## 概要

コード生成器の `expr_type()` が `Expr::Cast` ノードを処理していなかった致命的なバグを修正する。このバグにより、キャストを通じたデリファレンス（例: `*(int *)ptr`）で不正なメモリロードサイズが使用されていた。

## バグの内容

`*(int *)void_ptr` のようなコードがコンパイルされると、デリファレンスのコード生成は正しいロード命令を選択するためにデリファレンス先の型を知る必要がある:
- `int` → `movslq (%rax), %rax`（符号拡張付き4バイトロード）
- `long`/ポインタ → `mov (%rax), %rax`（8バイトロード）

`expr_type()` 関数は式ノードをたどって型を判定する。`Deref(Cast{ty: Ptr(Int), expr})` の場合:

1. `expr_type(Deref(inner))` がポインタ型を取得するために `expr_type(inner)` を呼び出す
2. `expr_type(Cast{ty: Ptr(Int), ...})` には専用のケースがなかった
3. `_ => Type::long_type()`（デフォルト）にフォールスルーした
4. Derefに戻ると、`Long` は `Ptr` でも `Array` でもないため、再び `Type::long_type()` にフォールスルーした
5. `emit_load_indirect(long)` が `mov (%rax), %rax` を出力 — 4バイトデータに対して8バイトロードが行われた

## 症状

8バイトロードは、対象変数の4バイトに加えて隣接するスタックデータ4バイト（通常は別の変数や保存された `%rbp`）を読み込む。これにより上位32ビットにゴミが入った64ビット値が生成される。

結果を一旦 `int` 変数に格納した場合は、`movl %eax, -offset(%rbp)` で32ビットに切り詰められるためバグが隠蔽される。しかし `if (func_call() > 0)` のように比較で直接使用すると、ゴミを含む64ビット値で比較が行われ、誤った結果が生じる。

例:
```c
int int_cmp(void *a, void *b) {
    return *(int *)a - *(int *)b;  // 8-byte loads → garbage in upper bits
}

int main() {
    int x = 10, y = 5;
    if (int_cmp(&x, &y) > 0) return 1;  // Failed: compared garbage 64-bit value
    return 0;
}
```

## 修正内容

`expr_type()` に `Expr::Cast` ケースを追加:

```rust
Expr::Cast { ty, .. } => ty.clone(),
```

これにより `expr_type(Cast{ty: Ptr(Int), ...})` は正しく `Ptr(Int)` を返し、`Deref` がベース型 `Int` を抽出して、`movslq (%rax), %rax`（正しい4バイト符号拡張ロード）が選択されるようになった。

## 修正前後の生成アセンブリ

修正前:
```asm
  mov (%rax), %rax     # 8-byte load — reads 4 bytes of garbage
```

修正後:
```asm
  movslq (%rax), %rax  # 4-byte load with sign extension — correct
```

## テストケース

```c
int main() { int a=10; int b=5; void *pa=&a; void *pb=&b; return *(int*)pa - *(int*)pb; }  // → 5
int int_cmp(void *a, void *b) { return *(int*)a - *(int*)b; }
int main() { int x=10; int y=5; if (int_cmp(&x, &y) > 0) return 1; return 0; }              // → 1
```
