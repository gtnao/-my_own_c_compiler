# Step 14.29: Fix Increment/Decrement for Any Lvalue + Forward-Declared Struct Members

## Overview

Two fixes in this step:
1. Generalize `++`/`--` (pre and post) to work with any lvalue expression, not just simple variables
2. Fix forward-declared struct resolution to also update struct member types

## Bug 1: `p->field++` Does Nothing

### The Problem

Pre/post increment/decrement operators only handled `Expr::Var(name)`:

```rust
Expr::PostInc(operand) => {
    if let Expr::Var(name) = operand.as_ref() {
        self.emit_load_var(name);
        // ...
    }
    // If operand isn't Var, silently does nothing!
}
```

So `l->length++` (where operand is `Expr::Member`) would silently be a no-op.

### The Fix

Replaced all four operators (PreInc, PreDec, PostInc, PostDec) with generic lvalue handling using `gen_addr()`, `emit_load_indirect()`, and `emit_store_indirect()`.

**PostInc algorithm:**
```
gen_addr(operand)      // address in %rax
push()                 // save address on stack
emit_load_indirect()   // load old value from (%rax) into %rax
mov %rax, %rcx         // save old value in %rcx
add $1, %rax           // compute new value
pop %rdi               // restore address into %rdi
emit_store_indirect()  // store new value (%rax) to (%rdi)
mov %rcx, %rax         // return old value (post-increment semantics)
```

**PreInc algorithm:**
```
gen_addr(operand)      // address in %rax
push()                 // save address
emit_load_indirect()   // load current value
add $1, %rax           // increment
pop %rdi               // restore address
emit_store_indirect()  // store new value
                       // %rax already has new value (pre-increment semantics)
```

### Key Insight: `gen_addr` Already Supports Complex Lvalues

`gen_addr()` handles:
- `Expr::Var(name)` — `lea -offset(%rbp), %rax` or `lea name(%rip), %rax`
- `Expr::Deref(inner)` — evaluates inner to get pointer value
- `Expr::Member(base, name)` — `gen_addr(base)` + member offset

So `p->field` (which is `Deref(Var("p")) → Member`) naturally computes the correct memory address.

### Register Usage

The approach uses `%rcx` as a temporary to hold the old value (for PostInc/PostDec). This is safe because:
- `%rcx` is a caller-saved register in System V AMD64 ABI
- `emit_store_indirect()` only uses `%rax` and `%rdi`
- No function calls happen between saving and restoring `%rcx`

## Bug 2: Forward-Declared Struct Members Not Updated

When a struct member contains a pointer to a forward-declared struct:

```c
struct Node; typedef struct Node Node;
struct Node { int val; Node *next; };  // next is Ptr(empty struct) until update
```

The `update_struct_members_with_struct()` function scans both `struct_tags` and `typedefs` to find struct types whose members reference the forward-declared struct, and updates them with the full definition.

## Test Cases

```c
// Post-increment through arrow
typedef struct { int length; } List;
List *l = malloc(sizeof(List)); memset(l, 0, sizeof(List));
l->length++; l->length++; l->length++;
return l->length;  // → 3

// Pre-increment through arrow
typedef struct { int val; } S;
S *p = malloc(sizeof(S)); memset(p, 0, sizeof(S));
++p->val; ++p->val;
return p->val;  // → 2

// Post-increment return value through arrow
p->val = 5; int old = p->val++;
return old * 10 + p->val;  // → 56 (old=5, new=6)

// Dereference + post-increment
int a[2]; a[0] = 3; int *p = a;
return (*p)++;  // → 3 (returns old value)

// Linked list with forward-declared struct
struct Node; typedef struct Node Node;
struct Node { int val; Node *next; };
return a->next->val;  // → 42
```
