# Step 1.4: 乗除算、優先順位、括弧、単項演算子

## 概要

コンパイラの中核アーキテクチャを導入するステップ。`ast.rs` (AST定義)、`parser.rs` (再帰下降パーサー)、`codegen.rs` (コード生成器) を新規作成し、演算子の優先順位、括弧、単項演算子を正しく処理する。

## 入出力

**入力**: `5 + 6 * 7`

**出力**:
```asm
  .globl main
main:
  push %rbp
  mov %rsp, %rbp
  mov $7, %rax
  push %rax
  mov $6, %rax
  pop %rdi
  imul %rdi, %rax
  push %rax
  mov $5, %rax
  pop %rdi
  add %rdi, %rax
  mov %rbp, %rsp
  pop %rbp
  ret
```

`5 + (6 * 7) = 47` が正しく計算される。

## アーキテクチャの変更

### コンパイラパイプライン

```
ソースコード → Lexer → トークン列 → Parser → AST → Codegen → アセンブリ
```

Step 1.3 までは `main.rs` がトークン列を直接走査してアセンブリを生成していたが、このステップからはASTを介した2段階の処理になる。

### 新規ファイル

| ファイル | 役割 |
|---------|------|
| `src/ast.rs` | AST (抽象構文木) のノード定義 |
| `src/parser.rs` | 再帰下降パーサー (トークン列 → AST) |
| `src/codegen.rs` | コード生成器 (AST → x86-64 アセンブリ) |

### main.rs の変更

```rust
// Before (Step 1.3): トークン列を直接走査
let tokens = lexer.tokenize();
// ... 手動でトークンを消費してprintln!

// After (Step 1.4): パイプライン
let tokens = lexer.tokenize();
let expr = Parser::new(tokens).parse();      // トークン列 → AST
let output = Codegen::new().generate(&expr); // AST → アセンブリ
print!("{}", output);
```

## ast.rs — 抽象構文木

```rust
pub enum BinOp { Add, Sub, Mul, Div }
pub enum UnaryOp { Neg }

pub enum Expr {
    Num(i64),
    BinOp { op: BinOp, lhs: Box<Expr>, rhs: Box<Expr> },
    UnaryOp { op: UnaryOp, operand: Box<Expr> },
}
```

ASTは式 (expression) の木構造。例えば `5 + 6 * 7` は:

```
    Add
   /   \
  5    Mul
      /   \
     6     7
```

`Box<Expr>` を使うのは、Rustでは再帰的な型は固定サイズにする必要があるため。`Box` はヒープアロケーションで間接参照にする。

## parser.rs — 再帰下降パーサー

### 文法規則

演算子の優先順位を文法規則で表現する:

```
expr    = mul ("+" mul | "-" mul)*
mul     = unary ("*" unary | "/" unary)*
unary   = ("+" | "-") unary | primary
primary = num | "(" expr ")"
```

**優先順位** (低い → 高い):
1. `+`, `-` (加減算)
2. `*`, `/` (乗除算)
3. 単項 `+`, `-`
4. 数値リテラル、括弧

### なぜこの文法で優先順位が実現されるのか

`expr` が `mul` を呼び、`mul` が `unary` を呼ぶ。下の規則ほど「結合力が強い」。

`5 + 6 * 7` をパースする場合:
1. `expr` が `mul` を呼ぶ → `mul` は `5` を返す
2. `expr` は `+` を見つける
3. `expr` が再度 `mul` を呼ぶ → `mul` は `6 * 7` を木にして返す
4. `expr` は `Add(5, Mul(6, 7))` を構築

`*` は `mul` の中で処理されるので、`+` より先に結合する。

### 再帰下降パーサーとは

各文法規則に対応する関数を用意し、関数の呼び出しで文法のネストを表現する方式。手書きで書きやすく、エラー報告も柔軟にできる。

```rust
fn expr(&mut self) -> Expr {
    let mut node = self.mul();        // まず mul を評価
    loop {
        match self.current().kind {
            TokenKind::Plus => {
                self.advance();
                let rhs = self.mul(); // 右辺も mul を評価
                node = Expr::BinOp { op: BinOp::Add, lhs: node, rhs };
            }
            // ...
            _ => break,
        }
    }
    node
}
```

## codegen.rs — スタックマシン方式のコード生成

### スタックマシンとは

二項演算の両辺を **スタック** を使って計算する方式:

1. 右辺を評価 → 結果が `%rax` に入る
2. `%rax` をスタックに push
3. 左辺を評価 → 結果が `%rax` に入る
4. スタックから pop して `%rdi` に入れる (= 右辺の値)
5. `%rax` (左辺) と `%rdi` (右辺) で演算

```rust
fn gen_expr(&mut self, expr: &Expr) {
    match expr {
        Expr::BinOp { op, lhs, rhs } => {
            self.gen_expr(rhs);          // rhs → %rax
            self.emit("  push %rax");    // save rhs
            self.gen_expr(lhs);          // lhs → %rax
            self.emit("  pop %rdi");     // rdi = rhs
            // now: %rax = lhs, %rdi = rhs
            match op {
                BinOp::Add => self.emit("  add %rdi, %rax"),
                BinOp::Sub => self.emit("  sub %rdi, %rax"),
                BinOp::Mul => self.emit("  imul %rdi, %rax"),
                BinOp::Div => {
                    self.emit("  cqto");
                    self.emit("  idiv %rdi");
                }
            }
        }
        // ...
    }
}
```

### x86-64 の主要レジスタ

CPU内部にある超高速な小さなメモリ。x86-64には汎用レジスタが16個あり、それぞれ64ビット（8バイト）。

| レジスタ | 役割 |
|---------|------|
| `%rax` | 関数の**戻り値**、計算の一時的な結果 |
| `%rsp` | **スタックポインタ** — スタックの一番上を常に指す |
| `%rbp` | **ベースポインタ（フレームポインタ）** — 関数のスタック領域の「基準点」 |
| `%rdi` | 関数の第1引数 / 二項演算の一時レジスタとして使用 |
| `%rdx` | 除算時の上位64ビット / 余りの格納先 |

### スタックとは

メモリ上に確保された「積み重ね」領域。**下方向に伸びる**のが重要なポイント。

```
高いアドレス  ┌─────────────┐
              │  ...        │
              ├─────────────┤
              │  戻りアドレス │ ← call命令が自動でpush
              ├─────────────┤
              │             │
低いアドレス  └─────────────┘ ← %rsp (スタックの先頭)
```

- `push %rax` → `%rsp` を8バイト**下げて**、そこに `%rax` を書き込む
- `pop %rax` → `%rsp` の位置から読み取って `%rax` に入れ、`%rsp` を8バイト**上げる**

### 新しいアセンブリ命令

| 命令 | 動作 | 使用場面 |
|------|------|---------|
| `push %rbp` | `%rbp` をスタックにpush | 関数プロローグ |
| `pop %rbp` | スタックトップを `%rbp` に pop | 関数エピローグ |
| `mov %rsp, %rbp` | スタックポインタをフレームポインタにコピー | 関数プロローグ |
| `imul %rdi, %rax` | `%rax = %rax * %rdi` (符号付き乗算) | 乗算 |
| `cqto` | `%rax` を `%rdx:%rax` に符号拡張 | 除算の前準備 |
| `idiv %rdi` | `%rdx:%rax / %rdi` → 商=`%rax`, 余り=`%rdx` | 除算 |
| `neg %rax` | `%rax = -%rax` (二の補数の否定) | 単項マイナス |

### 関数プロローグ/エピローグ

```asm
main:
  push %rbp          # ① 呼び出し元の rbp を退避（壊さないため）
  mov %rsp, %rbp     # ② rbp を基準点として固定
  ...                 # ③ 本体（push/pop で %rsp が動いても rbp は不変）
  mov %rbp, %rsp     # ④ %rsp を基準点に戻す（ゴミを一掃）
  pop %rbp           # ⑤ 呼び出し元の rbp を復元
  ret                 # ⑥ 戻りアドレスにジャンプ
```

#### スタックの変化を図で追う

**① `main()` が呼ばれた直後** — `call main` がリターンアドレスを自動でpush:

```
              ┌─────────────┐
              │ 戻りアドレス  │ ← %rsp
              └─────────────┘

%rbp = 呼び出し元の値（まだ触ってない）
```

**② `push %rbp`** — 呼び出し元の rbp を保存:

```
              ┌─────────────┐
              │ 戻りアドレス  │
              ├─────────────┤
              │ 古い rbp     │ ← %rsp
              └─────────────┘
```

なぜ保存するか: `%rbp` は呼び出し元の関数も使っている。壊さずに後で元に戻すため。

**③ `mov %rsp, %rbp`** — フレームポインタを設定:

```
              ┌─────────────┐
              │ 戻りアドレス  │
              ├─────────────┤
              │ 古い rbp     │ ← %rsp, %rbp 両方ここを指す
              └─────────────┘
```

`%rbp` を「この関数のスタック領域の基準点」として固定する。以降、`%rsp` が push/pop で動いても、`%rbp` は変わらない。

**なぜ基準点が必要か**: 後のステップ (Step 2.2) でローカル変数を追加すると、変数は `%rbp` からの相対位置でアクセスする（例: `%rbp-8` が変数a、`%rbp-16` が変数b）。`%rsp` は計算中に動くので基準点にできない。

**④ 関数の本体が実行される** — 計算中に push/pop で `%rsp` は上下に動く:

```
              ┌─────────────┐
              │ 戻りアドレス  │
              ├─────────────┤
              │ 古い rbp     │ ← %rbp（固定）
              ├─────────────┤
              │ 計算用の一時値 │ ← %rsp（動く）
              └─────────────┘
```

**⑤ `mov %rbp, %rsp`** — スタックを巻き戻す:

```
              ┌─────────────┐
              │ 戻りアドレス  │
              ├─────────────┤
              │ 古い rbp     │ ← %rsp = %rbp に戻る
              └─────────────┘
```

計算中にpushしたゴミがどれだけあっても、`%rsp` を `%rbp` の位置に戻すだけで全部捨てられる。

**⑥ `pop %rbp`** — 呼び出し元の rbp を復元:

```
              ┌─────────────┐
              │ 戻りアドレス  │ ← %rsp
              └─────────────┘

%rbp = 呼び出し元の元の値に復元された
```

**⑦ `ret`** — 呼び出し元に戻る。`ret` = `pop %rip`（命令ポインタに戻りアドレスをセット）。スタックは呼ばれる前の状態に完全復元される。

この仕組みがあるから、関数が何段ネストしても各関数のスタック領域が正しく管理される。Step 3.2 で再帰（`fact(5)` → `fact(4)` → ... → `fact(1)`）が動くのもこの仕組みのおかげ。

### 除算の仕組み — cqto と idiv の詳細

#### なぜ 128ビットなのか

x86-64 の `idiv` 命令は歴史的な設計で、**常に2つのレジスタをペアで被除数として使う**:

```
%rdx : %rax  (128ビット)  ÷  %rdi (64ビット)
─────────────────────────────────────────────
商 → %rax    余り → %rdx
```

`10 / 3` を計算したいとき、被除数の `10` は64ビットの値だが、`idiv` は `%rdx:%rax` の128ビットを割ろうとする。つまり **`%rdx` の中身も被除数の一部として扱ってしまう**。

#### %rdx にゴミが入っていると壊れる

もし `%rdx` にゴミ（例えば `0xFFFF...`）が残っていたら:

```
%rdx = 0xFFFFFFFFFFFFFFFF
%rax = 0x000000000000000A  (10)

%rdx:%rax = 0xFFFFFFFFFFFFFFFF_000000000000000A
          = とんでもなく大きな数

これを %rdi = 3 で割ると → 正しい結果にならない（または例外発生）
```

#### cqto が何をするか

`cqto` (Convert Quad to Octo) は `%rax` の **最上位ビット (符号ビット)** を `%rdx` の全64ビットにコピーする。

**正の数の場合** (例: `%rax = 10`):

```
%rax = 0x000000000000000A
       ^
       符号ビット = 0 (正)

cqto 実行後:
  %rdx = 0x0000000000000000  (全ビット0で埋める)
  %rax = 0x000000000000000A  (変化なし)

  %rdx:%rax = 0x0000000000000000_000000000000000A = 10
```

**負の数の場合** (例: `%rax = -10`):

二の補数表現では負の数は最上位ビットが1:

```
%rax = 0xFFFFFFFFFFFFFFF6  (-10 の二の補数)
       ^
       符号ビット = 1 (負)

cqto 実行後:
  %rdx = 0xFFFFFFFFFFFFFFFF  (全ビット1で埋める)
  %rax = 0xFFFFFFFFFFFFFFF6  (変化なし)

  %rdx:%rax = 0xFFFFFFFFFFFFFFFF_FFFFFFFFFFFFFFF6 = -10 (128ビットとしても-10)
```

#### なぜ「符号拡張」と呼ぶのか

8ビットの `-10` を16ビットに広げる場合で考えるとわかりやすい:

```
 8ビット:  0xF6    = 1111 0110  (-10)
16ビット:  0xFFF6  = 1111 1111 1111 0110  (-10)
                      ^^^^^^^^^
                      符号ビット(1)で左側を埋めた
```

上位ビットを符号ビットで埋めれば、**数としての値は変わらずにビット幅だけ広がる**。これが符号拡張。`cqto` は同じことを 64ビット → 128ビット で行う。

#### 除算の全体の流れ (10 / 3 の場合)

```asm
  # この時点で %rax = 10 (左辺), %rdi = 3 (右辺)

  cqto            # %rdx = 0, %rax = 10  → %rdx:%rax = 10
  idiv %rdi       # 10 ÷ 3 = 商3, 余り1
                  # → %rax = 3 (商)
                  # → %rdx = 1 (余り)
```

除算 (`/`) の場合はそのまま `%rax` (商) を使う。
剰余 (`%`) の場合は `mov %rdx, %rax` で余りを `%rax` にコピーする (Step 1.5 で実装)。

#### cqto の名前の由来

| 名前 | 意味 | 変換 |
|------|------|------|
| `cqto` | Convert **Quad** to **Octo** | 4ワード(64bit) → 8ワード(128bit) |

x86用語では1ワード = 16ビット。Quad = 4ワード = 64ビット、Octo = 8ワード = 128ビット。Intel構文では同じ命令を `cqo` と書く。
