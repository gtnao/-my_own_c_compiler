# Step 14.10: `__extension__` Keyword

## Overview

Add support for the GCC `__extension__` keyword. It is consumed and ignored wherever it appears — before declarations or expressions.

## Why This Matters

GCC uses `__extension__` to suppress warnings about non-standard extensions. PostgreSQL and system headers use it in macro definitions:

```c
__extension__ typedef unsigned long long __uint64_t;
__extension__ ({ int __val = expr; __val; })
```

## Implementation

### Token

Added `Extension` variant to `TokenKind`.

### Lexer

Recognizes `__extension__` as a keyword, mapping to `TokenKind::Extension`.

### Parser

`__extension__` is consumed and ignored in two locations:

1. **Before type declarations** — alongside `inline`, `_Noreturn`, and `register` in `parse_type()`:
   ```rust
   while matches!(self.current().kind,
       TokenKind::Inline | TokenKind::Noreturn | TokenKind::Register | TokenKind::Extension) {
       self.advance();
   }
   ```

2. **Before expressions** — at the start of `unary()`:
   ```rust
   while self.current().kind == TokenKind::Extension {
       self.advance();
   }
   ```

Added to `is_type_keyword()` and `stmt()` type-start patterns.

## Behavior

- `__extension__` is a GCC-specific keyword that suppresses warnings about language extensions
- Our compiler has no warnings to suppress, so it is simply consumed and ignored
- Works before both declarations (`__extension__ int a = 5;`) and expressions (`return __extension__ 3;`)

## Test Cases

```c
int main() { __extension__ int a = 5; return a; }  // → 5
int main() { return __extension__ 3; }              // → 3
```
