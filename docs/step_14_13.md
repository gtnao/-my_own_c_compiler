# Step 14.13: `_Static_assert`

## Overview

Add support for `_Static_assert(expression, "message")` declarations. These are consumed and ignored — no compile-time assertion checking is performed.

## Why This Matters

PostgreSQL uses `_Static_assert` (via the `StaticAssertStmt` macro) to verify compile-time invariants:

```c
StaticAssertStmt(sizeof(FormData_pg_attribute) == ATTRIBUTE_FIXED_PART_SIZE,
    "wrong pg_attribute size");
_Static_assert(sizeof(int) == 4, "int must be 4 bytes");
```

## Implementation

### Token

Added `StaticAssert` variant to `TokenKind`.

### Lexer

Recognizes `_Static_assert` as a keyword, mapping to `TokenKind::StaticAssert`.

### Parser

Added `skip_static_assert()` helper method that:
1. Consumes the `_Static_assert` token
2. Consumes `(`
3. Skips all tokens until the matching `)` (tracking paren depth for nested expressions)
4. Consumes optional trailing `;`

Called in two locations:
- **Top-level `parse()`** — before function/variable parsing
- **`stmt()`** — as a statement that produces an empty block

```rust
fn skip_static_assert(&mut self) {
    self.advance(); // _Static_assert
    self.expect(TokenKind::LParen);
    let mut depth = 1;
    while depth > 0 {
        match self.current().kind {
            TokenKind::LParen => depth += 1,
            TokenKind::RParen => depth -= 1,
            TokenKind::Eof => break,
            _ => {}
        }
        self.advance();
    }
    if self.current().kind == TokenKind::Semicolon {
        self.advance();
    }
}
```

## Behavior

- The assertion expression and message string are completely ignored
- No compile-time evaluation or error checking is performed
- The declaration produces no code output
- Both top-level and block-level `_Static_assert` are supported

## Test Cases

```c
_Static_assert(1, "ok"); int main() { return 1; }                         // → 1
int main() { _Static_assert(sizeof(int) == 4, "int is 4"); return 5; }    // → 5
_Static_assert(1, "a"); _Static_assert(1, "b"); int main() { return 3; }  // → 3
```
