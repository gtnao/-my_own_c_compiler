# ステップ 14.13: `_Static_assert`

## 概要

`_Static_assert(expression, "message")` 宣言のサポートを追加する。読み取られた後、無視される。コンパイル時のアサーションチェックは行わない。

## なぜ必要か

PostgreSQL は（`StaticAssertStmt` マクロを通じて）コンパイル時の不変条件を検証するために `_Static_assert` を使用している:

```c
StaticAssertStmt(sizeof(FormData_pg_attribute) == ATTRIBUTE_FIXED_PART_SIZE,
    "wrong pg_attribute size");
_Static_assert(sizeof(int) == 4, "int must be 4 bytes");
```

## 実装

### トークン

`TokenKind` に `StaticAssert` バリアントを追加した。

### 字句解析

`_Static_assert` をキーワードとして認識し、`TokenKind::StaticAssert` にマッピングする。

### 構文解析

`skip_static_assert()` ヘルパーメソッドを追加した。このメソッドは以下を行う:
1. `_Static_assert` トークンを読み取る
2. `(` を読み取る
3. 対応する `)` まですべてのトークンをスキップする（ネストされた式のために括弧の深さを追跡する）
4. オプションの末尾の `;` を読み取る

2箇所で呼び出される:
- **トップレベルの `parse()`** — 関数/変数の解析の前
- **`stmt()`** — 空のブロックを生成する文として

```rust
fn skip_static_assert(&mut self) {
    self.advance(); // _Static_assert
    self.expect(TokenKind::LParen);
    let mut depth = 1;
    while depth > 0 {
        match self.current().kind {
            TokenKind::LParen => depth += 1,
            TokenKind::RParen => depth -= 1,
            TokenKind::Eof => break,
            _ => {}
        }
        self.advance();
    }
    if self.current().kind == TokenKind::Semicolon {
        self.advance();
    }
}
```

## 動作

- アサーション式とメッセージ文字列は完全に無視される
- コンパイル時の評価やエラーチェックは行わない
- この宣言はコード出力を生成しない
- トップレベルとブロックレベルの両方の `_Static_assert` に対応している

## テストケース

```c
_Static_assert(1, "ok"); int main() { return 1; }                         // → 1
int main() { _Static_assert(sizeof(int) == 4, "int is 4"); return 5; }    // → 5
_Static_assert(1, "a"); _Static_assert(1, "b"); int main() { return 3; }  // → 3
```
