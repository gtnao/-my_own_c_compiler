# Step 14.17: `void *` Pointer Arithmetic and Declarations

## Overview

Fix `void *` variable declarations inside function bodies and confirm `void *` pointer arithmetic works correctly (GCC extension: treating `sizeof(void)` as 1 byte for arithmetic purposes).

## Why This Matters

PostgreSQL uses `void *` extensively for generic pointer manipulation:

```c
void *ptr = palloc(size);
ptr = (char *)ptr + offset;  // or just ptr + offset with GCC extension
memcpy(dest, src, len);
```

## Implementation

### Parser Fix

The `stmt()` match pattern was missing `TokenKind::Void`, causing `void *p = ...;` declarations inside function bodies to fail with "expected a number, identifier or '('". Added `TokenKind::Void` to the type-start pattern in `stmt()`:

```rust
TokenKind::Void | TokenKind::Int | TokenKind::Char | ... => {
    self.var_decl()
}
```

### Void Pointer Arithmetic

The `void *` pointer arithmetic already works correctly due to how the codegen handles pointer scaling:

```rust
if lhs_ty.is_pointer() {
    let size = lhs_ty.base_type().unwrap().size();
    if size > 1 {
        self.emit(&format!("  imul ${}, %rdi", size));
    }
}
```

Since `TypeKind::Void` has `size() == 0`, the condition `size > 1` is false, so no scaling is applied. This means `void *p + n` advances by `n` bytes — which matches GCC's behavior where `sizeof(void) == 1` for pointer arithmetic.

## Behavior

- `void *p = &a;` — valid declaration of void pointer
- `p + 4` — advances by 4 bytes (no scaling, GCC extension behavior)
- `*(int *)p` — cast void pointer to typed pointer and dereference

## Test Cases

```c
int main() { int a[2]; a[0]=10; a[1]=20; void *p=&a[0]; p=p+4; return *(int*)p; }  // → 20
int main() { int a = 10; void *p = &a; return *(int*)p; }                            // → 10
```
